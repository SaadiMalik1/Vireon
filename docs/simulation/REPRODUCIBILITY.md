# VIREON Research Reproducibility Standards

This document establishes the strict criteria required for any simulation, benchmark, or report generated by the VIREON ecosystem to be considered scientifically reproducible and academically valid.

## Core Reproducibility Axioms

1. **Absolute Determinism:** Given the same initial `StateStore` configuration, identically loaded plugins, and the same master cryptographic seed, the engine MUST produce a byte-for-byte identical output trace regardless of the host operating system or hardware architecture.
2. **Provenance Tracking:** Every generated telemetry file must include a cryptographic header detailing the exact Git commit hashes of `vireon` and `neurodsl`, the master seed, and the SHA-256 hashes of all active plugins.
3. **No Unseeded Randomness:** The use of `random`, `numpy.random`, or `rand` (Rust) without explicitly deriving the RNG from the VIREON orchestrator's master seed is strictly prohibited and will cause CI to fail.

## Implementation Roadmap

### 1. The Global Seed Manager
The `SimulationBuilder` must be refactored to accept a `master_seed`. This seed will be used to initialize a `SeedManager` that dispenses deterministic sub-seeds to all initialized plugins, physics providers, and the `neurodsl` Rust engine.

### 2. Output Checksumming
The `vireon.runtime` layer must automatically compute a rolling SHA-256 checksum of the `EventBus` message stream. At the end of a simulation, this checksum acts as the absolute proof of execution fidelity.

### 3. Floating Point Determinism
- **Rust (`neurodsl`):** Ensure that FFT implementations and ODE solvers do not utilize fast-math compiler flags that compromise IEEE 754 floating-point determinism across different CPU architectures (e.g., x86 vs ARM).
- **Python (`vireon`):** Ensure `numpy` utilizes deterministic BLAS/LAPACK backends, or restrict mathematically sensitive operations entirely to the Rust engine.

### 4. Continuous Reproducibility CI
Add a CI step that runs a 10-second reference simulation on both an `ubuntu-latest` (x86_64) runner and a `macos-latest` (arm64) runner. The step MUST fail if the resulting telemetry hashes differ.
